# Global Roll-up Formats: `global-backlog.md` and `global-maturity.md`

Both `~/.claude/global-backlog.md` and `~/.claude/global-maturity.md` are
auto-regenerated by `portfolio rebuild` from per-project sources — the
registry, each project's `docs/backlog.md`, and each project's
`docs/MATURITY.md`. Neither file should ever be edited directly except for
one section: the `## Cross-project items` block in `global-backlog.md`. That
section is the only hand-edited content in either file and is preserved
byte-for-byte across every rebuild. Everything outside it is overwritten.

## ~/.claude/global-backlog.md template

```
# Global Backlog

Auto-generated index of every per-project `docs/backlog.md`. Edit the
`## Cross-project items` section below by hand; everything else is
regenerated on `portfolio rebuild`.

**Last rebuilt:** <YYYY-MM-DD HH:MM>

---

## Per-project backlogs

<repeated block, one per registered+enabled project that has docs/backlog.md>
### <area>/<name> — <COUNT> open
- **Path:** `<absolute path>/docs/backlog.md`
- **3 newest:** <comma-sep top-3 BL-NNN titles or "none">
</repeated block>

---

## Cross-project items

<!-- BEGIN PRESERVE — content below this line is preserved across rebuilds -->

<user-curated items go here; format mirrors per-project BL-NNN style but
 uses GBL-NNN ids (next id = max(existing GBL-NNN)+1, never reuse)>

<!-- END PRESERVE -->
```

### Preservation rule

On rebuild, the orchestrator applies the following procedure:

1. Read the existing `~/.claude/global-backlog.md` in full.
2. Locate the sentinel pair:
   - Start sentinel: `<!-- BEGIN PRESERVE — content below this line is preserved across rebuilds -->`
   - End sentinel: `<!-- END PRESERVE -->`
3. Extract everything between the two sentinels (exclusive of the sentinel
   lines themselves) and hold it as the preserved block.
4. Regenerate the file header and `## Per-project backlogs` section from
   the current registry and per-project backlog files.
5. Write the new file, inserting the preserved block back between the
   sentinels, byte-for-byte.
6. If either sentinel is absent (first run, or the user accidentally
   removed them), insert an empty preserved block between freshly written
   sentinels and emit a one-line warning to the terminal:
   `WARNING: global-backlog.md preserve sentinels missing — inserted empty block`.
   Never attempt to guess or reconstruct lost content.

The sentinels are HTML-comment style (`<!-- ... -->`) so they are invisible
in rendered markdown but unambiguous in raw text processing.

## ~/.claude/global-maturity.md template

```
# Global Maturity Dashboard

Auto-generated from per-project `docs/MATURITY.md` files. Re-run
`portfolio maturity` after any per-project change.

**Last rebuilt:** <YYYY-MM-DD HH:MM>

---

| Project | Docs | Sec | Pkg | UI/UX | i18n | Tests/CI | Ship-ready? |
|---------|------|-----|-----|-------|------|----------|-------------|
| <area>/<name> | <cell> | <cell> | <cell> | <cell> | <cell> | <cell> | <yes\|no> |
...

---

## Legend

- 🟢 `auto`  — auto-detected, evidence linked from per-project MATURITY.md
- 🟡 `claim` — manually claimed (yellowed if claim is >90 days old)
- ⚪ `N/A`   — not applicable for this project
- 🔴 `–`     — axis not yet ticked
- ❓ `?`     — auto-detector failed; previous value retained

## ship_ready rule

A project is ship-ready iff every axis meets its per-axis threshold
(see references/maturity-axes.md → "## ship_ready aggregation").
```

### Cell rendering rule

Each cell shows the highest tick state present across the axis's sub-items,
rendered as an emoji plus a short fraction (e.g. `🟢 4/4` for an all-green
Documentation axis, `🟡 2/3` for a partially manually-claimed axis, `🔴 0/4`
for an axis with no ticks at all). The precedence order for the highest state
is: `auto` > `claim` > `N/A` > `–` > `?`. A `?` state (detector failed)
propagates as ❓ only when no higher state exists for that axis.

## Cross-project ID convention

Cross-project items in `global-backlog.md` use the prefix `GBL-` (Global
BackLog) with three-digit zero-padded numbering, e.g. `GBL-001`, `GBL-002`.
Rules:

- The next ID on any given write is `max(existing GBL-NNN) + 1`. If no
  GBL-NNN entries exist yet, start at `GBL-001`.
- IDs are never reused. Removing a GBL-NNN entry does not free its number.
- The format otherwise mirrors the per-project `BL-NNN` style defined in
  the `backlog` skill: same field set (`Title`, `Added`, `Source`, etc.),
  same markdown layout, different prefix.
- GBL entries are only valid inside the `## Cross-project items` preserve
  block. Do not write GBL-NNN identifiers into per-project `docs/backlog.md`
  files.

## Hard rules

- **Rebuilds are idempotent.** The same registry state and the same
  per-project file contents must produce byte-identical output. Timestamp
  handling: render a new `**Last rebuilt:**` value only when the generated
  content (excluding the timestamp line itself) differs from the prior file.
  If nothing changed, retain the prior timestamp and write nothing.
- **Never delete the PRESERVE block or its sentinels.** If the sentinel
  lines are missing on read, insert them with an empty body and log a
  warning. Do not silently drop previously curated GBL items.
- **Per-project section is sorted by `area/name` ascending** (lexicographic,
  case-insensitive). The sort is stable so projects in the same area sort
  alphabetically by name.
- **Projects with `enabled: false` in the registry are excluded from both
  files.** They appear in the registry for audit and toggling purposes but
  contribute no rows or blocks to either global roll-up.
- **Projects with no `docs/backlog.md` are excluded from
  `global-backlog.md`** (no `### area/name` block written for them), but
  they DO appear as a row in `global-maturity.md` if they have a
  `docs/MATURITY.md`. A project with neither file contributes nothing to
  either global file.
